去年年底，OpenAI 发布 deepresearch 的时候，一种很常见的评价是："以后做研究的人要失业了。"类似的焦虑每出一款 Agent 产品就会重演一遍：Claude Code 出来，程序员慌；豆包手机出来，运营慌。

但这种"替代焦虑"可能搞错了方向。人与 Agent 的关系，不是一条从"人做"到"AI 做"的单行道。更准确的图景，是这两个端点之间正在展开一个越来越丰富的中间地带——而这个中间地带，才是一线从业者真正在建设的东西。

## 两个端点：AutoResearch 和 MetaHarness

AutoResearch 代表 Agent 的"极端自主"：给定一个问题，Agent 自己去搜索、阅读、推理、综合，最后交付一份完整的研究报告或决策建议。整个过程人只需要提供一个初始 question，剩下的全是 Agent 的工作。

MetaHarness 代表人对 Agent 的"极端管控"：不是告诉 Agent 做什么，而是定义 Agent 能做什么、不能做什么、按什么节奏做、做到什么程度停。它是 Agent 之外的元层——通过 Prompt Engineering 控制认知框架，通过工具约束控制能力边界，通过工作流编排控制行为路径。

两者看起来是一个光谱的两端：一边是 Agent 的自主权最大化，一边是人的控制权最大化。但实际工程中，这两端从来不是对立的。

## 中间地带才是主战场

AutoResearch 如果完全脱离人的管控，交付出来的东西大概率没法直接用。它会引用错误的来源、遗漏关键的上下文、在不确定的地方给出自信的结论。真正的 deepresearch 产品，实际上在输出环节加了大量的 Harness 工作：结构化模板、来源标注、不确定性声明、分步可审查的输出格式。

反过来，MetaHarness 也不是要把 Agent 管死。好的 Harness 设计追求的恰恰是"给 Agent 足够的安全空间让它自己发挥"——就像给一个初级研究员划定课题范围、提供文献库、定义报告格式之后，让他自己去做。你不需要盯着他每一步，但你知道他走不出你画的圈。

所以有意思的事情不是站在 AutoResearch 那边喊"全自动"，也不是站在 MetaHarness 那边喊"全管控"。有意思的是中间那些具体的问题：

- 哪些步骤必须由人确认（比如付款、发邮件、改线上配置），哪些可以 Agent 自己决定？
- 人给的 feedback 应该是什么粒度——每一步给 yes/no，还是只给最终结果打分？
- Agent 失败了应该退回到哪一步？重新规划、换个工具、还是交还给人？
- 人离开 Agent 多久不干预是合理的——五分钟、半小时、还是一整天？

## 角色演变：从操作员到导演

在这些中间地带里，人的角色在发生一个微妙但根本的转变。

传统软件时代，人是操作员：你点击、输入、确认，软件执行。每一步都是人驱动的。

LLM 时代早期，人是指挥官：你给一个 prompt，模型给一个输出。人负责"说什么"，模型负责"怎么说"。

Agent 时代，人是导演。你不再指挥单个动作，而是设定场景、确定角色、给出预算和约束，然后让 Agent 自己去演。你干预的粒度从"每一步"变成了"关键决策点"。

这个转变对人的要求其实更高了，而不是更低。操作员需要的是执行力，指挥官需要的是表达能力，导演需要的是判断力、审美和风险意识。你要能看出来 Agent 什么地方可能在胡说、什么时候在磨蹭、什么决定一旦做错就很难挽回。

## Harness 不该是枷锁

有一种对 MetaHarness 的误解是把它当成"给 Agent 套上枷锁"。实际上，好的 Harness 更像"给 Agent 搭建舞台"。

一个没有舞台的演员只能尴尬地待在后台；一个舞台太大、没有边界的演员又会不知所措。Harness 的工作就是划定舞台大小、灯光方向、入场和退场时间——让 Agent 在它擅长的地方尽情发挥，在它不擅长的地方有明确的安全网。

三层的 Harness 体系恰好对应了三种"搭舞台"的方式：

- **Prompt Engineering** 搭的是认知的舞台：告诉 Agent 这个场景里什么是好的、什么是错的、按什么价值观做判断。
- **工具约束** 搭的是能力的舞台：给你这些工具，其他的不让碰。你能做的就这么多，但在这范围内随便发挥。
- **工作流编排** 搭的是行为的舞台：先做这个、再做那个，到这里必须等人确认，过了这个节点才能继续。

三层合在一起，既不是放任自流，也不是把 Agent 当木偶。它更像是把 Agent 当成一个需要培养的人：先教规矩，再给空间，关键时刻把关。

## 未来的关系图景

如果把 AutoResearch 到 MetaHarness 这条光谱铺开，我猜未来几年会出现几种不同的人-Agent 关系模式：

**审批者模式**：Agent 做大部分工作，人在关键节点审批。适合高风险、不可逆的操作场景。典型如金融交易、医疗诊断。

**协作者模式**：人和 Agent 交叉作业，互相补充。人写大纲 Agent 填内容，Agent 出初稿人修改，循环迭代。适合创作、编程、设计。

**导演模式**：人设定目标和约束，Agent 自主执行。人只在结果偏差或遇到异常时介入。适合信息检索、数据采集、批量处理。

**教练模式**：人通过 feedback 持续修正 Agent 的行为偏好，Agent 在实践中自我进化。适合个性化服务、长期陪跑型任务。

这些模式不是互斥的，同一个系统里不同任务可能用不同模式。核心是：**关系不是由 Agent 的能力决定的，而是由人对风险的判断决定的。** 一个能全自动完成的任务，如果后果很严重，人依然会选择审批者模式。一个只能半自动的任务，如果后果无伤大雅，人也可能放手让 Agent 自己跑。

## 结语

人类造工具的历史，有一个反复出现的规律：每一样新工具刚诞生的时候，人都担心被替代；但最终沉淀下来的，不是工具替代了人，而是工具重新定义了人的价值。

AutoResearch 不是终点，它只是把认知劳动的"自动化焦虑"推到了一个新高度。而 MetaHarness 提醒我们，在焦虑之外还有一件更重要的事：我们如何设计人与 Agent 之间的界面。这个界面决定了 Agent 是人的替代者还是人的延伸。

好的界面让人变得更像"人"——更专注于判断、审美、价值和方向，把那些可以规范化的、可以反复试错的、可以给 Agent 在划定范围内自由发挥的工作交出去。

这才是从 AutoResearch 到 MetaHarness 这条路上真正让人兴奋的东西。
