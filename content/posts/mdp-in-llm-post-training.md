## 核心问题

当前 LLM 后训练中的强化学习主要针对**逐字（token-level）预测**建模为 MDP，但现实中存在大量**多轮推理决策**场景（Agent 执行、工具调用链、对话规划），这类场景需要不同的 MDP 建模方式。

---

## 一、经典范式：Token-Level MDP

### 1.1 形式化定义

在 RLHF / PPO / GRPO 等主流后训练方法中，MDP 定义如下：

- **状态** $s_t$：当前上下文（prompt + 已生成的 tokens）
- **动作** $a_t$：下一个 token（词表大小 $|V|$ 的离散动作空间）
- **策略** $\pi_\theta(a_t | s_t)$：语言模型本身
- **奖励** $r_t$：通常是**稀疏的**，只在序列末尾给出（如人类偏好评分、答案正确性）
- **状态转移**：确定性，$s_{t+1} = s_t \oplus a_t$（拼接新 token）

```
[Prompt] → token₁ → token₂ → ... → tokenₙ → [Reward]
              ↑         ↑               ↑
           action₁   action₂        actionₙ
```

### 1.2 核心特点与工程处理

- **奖励极稀疏**：序列可能有数千步，奖励只在最后给出
- **动作空间极大**：$|V| \approx 10^4 \sim 10^5$

工程上的常见处理：
- **GAE**：训练 Critic 网络估计每个位置的 value，做广义优势估计
- **GRPO**：用组内相对奖励代替绝对奖励，去掉 Critic
- **DPO**：完全绕开 RL，直接在偏好数据上做监督，把 token-level MDP 转化为序列级对比学习

### 1.3 根本假设及其局限

Token-level MDP 隐含假设：**生成质量可以被分解为每步 token 选择的质量之和**。

这个假设在单轮问答、代码生成、文本摘要等**静态生成任务**中是合理的近似，但在**动态交互任务**中会导致优化目标与真实目标脱节。

## 二、新范式：Step-Level MDP

### 2.1 为什么需要更粗粒度的建模？

当 LLM 作为 Agent 执行多步任务时，决策单元不再是 token，而是：
- 一次工具调用（Tool Call）
- 一次推理步骤（Chain-of-Thought step）
- 一轮对话（turn）
- 一个子任务（Task decomposition step）

此时 token-level MDP 的问题暴露：
- **信用分配更难**：一个错误的工具调用可能在 100 个 token 后才体现出来
- **奖励信号粒度不匹配**：任务成功/失败是任务级的，不是 token 级的
- **策略优化目标错误**：优化每个 token 的概率 ≠ 优化任务完成率

### 2.2 两层 MDP 的嵌套关系

```
Step-Level MDP（粗粒度）
  S₀ ──A₀──> S₁ ──A₁──> S₂ ──...──> Sₜ
  ↓           ↓           ↓
  每个 Aₖ 对应一次 Tool Call 或 Reasoning Step

Token-Level MDP（细粒度，嵌套在每个 Aₖ 的生成中）
  Aₖ = token₁ + token₂ + ... + tokenₘ
```

这实际上是一个**层级 MDP（Hierarchical MDP）** 或 **Semi-MDP（半马尔科夫决策过程）** 的结构：
- 高层策略：决定"做什么"（选择工具/行动类型）
- 低层策略：决定"怎么说"（生成具体的 token 序列）

## 三、多轮推理决策中的核心挑战

### 3.1 信用分配问题

**现有方案对比**：

| | ORM（结果奖励） | PRM（过程奖励） |
|---|---|---|
| **奖励来源** | 最终结果正确性 | 每步推理质量 |
| **数据成本** | 低（只需标注答案） | 高（需标注每步） |
| **信用分配** | 不解决，靠算法处理 | 直接提供 |
| **泛化性** | 依赖最终任务 | 可迁移到新任务 |
| **典型应用** | RLHF、DPO | AlphaCode、Math Olympiad |

### 3.2 马尔科夫性假设的违背

经典 MDP 假设当前状态包含所有必要信息，但在 Agent 场景中经常被违背：
- 工具调用的副作用可能影响未来状态（如修改了数据库）
- 用户意图随对话进化，无法从单次状态还原
- 外部环境的变化（API 限流、网络故障）

**解法**：将历史全部纳入"状态"定义（POMDP，或直接用完整上下文作为状态）

### 3.3 探索-利用困境的特殊性

Token-level RL 中的探索：对 token 概率分布加温度/噪声。

Agent-level RL 中的探索更复杂：
- **工具选择的探索**：尝试不同的工具组合
- **策略路径的探索**：不同的任务分解方式
- **风险不对称**：某些探索（如删除文件）代价极高

**关键洞察**：Agent 的探索策略需要感知风险，不能像 token-level 那样均匀探索。

## 四、当前前沿方向

### 4.1 基于轨迹的优化（RLVR / STaR）

核心思路：
1. 让模型生成多条解题轨迹
2. 用验证器（代码执行结果、数学答案正确性）筛选正确轨迹
3. 用正确轨迹做监督训练或 RL 优化

**优点**：不需要 PRM，直接利用结果反馈
**局限**：只适用于有客观验证标准的任务

### 4.2 MCTS 与 LLM 的结合（AlphaCode 2、rStar）

用蒙特卡洛树搜索在推理步骤空间中做规划，用价值网络评估中间状态。

```
根节点（初始状态）
├── 推理路径 A
│   ├── Step A1 → (value: 0.8)
│   └── Step A2 → (value: 0.3)
└── 推理路径 B
    └── Step B2 → (value: 0.9) ← 选择
```

**挑战**：价值网络的训练本身需要大量轨迹数据

### 4.3 Options Framework（选项框架）

将一段连续动作抽象为一个"选项"（option），在选项层面做 RL，在 token 层面做监督学习。

这对应 LLM Agent 的自然分层：
- **高层**：规划层，决定执行哪些子任务（选项选择）
- **低层**：执行层，具体生成每步的文本（token 生成）

## 五、核心观点与结论

### 不同任务的 MDP 粒度选择

| 任务类型 | 推荐 MDP 粒度 | 奖励类型 |
|---------|------------|---------|
| 单轮问答/写作 | Token-level | ORM（结果） |
| 数学推理/代码生成 | Step-level（思维链步骤） | PRM + ORM |
| 工具调用 Agent | Tool-call-level | 任务完成率 |
| 多轮对话 | Turn-level | 对话满意度 |
| 复杂任务规划 | Task-level（子任务） | 层级奖励 |

### 统一框架：层级强化学习（HRL）

**层级强化学习** 提供了统一视角：
- 不同粒度的 MDP 是同一个问题的不同抽象层次
- 高层策略产生低层策略的目标，低层策略产生高层策略的状态转移
- LLM 的天然层级（规划 → 推理 → 生成）与 HRL 的层级结构高度吻合

**核心待解问题**：如何在不同层级之间高效传递梯度信号？

## 六、开放问题

1. **奖励设计**：多步任务中，中间奖励如何设计才不会引入"奖励欺骗"（reward hacking）？
2. **off-policy 数据利用**：Agent 的历史轨迹如何用于 off-policy 训练？
3. **安全约束**：如何在 RL 框架内加入硬约束（不能执行某些危险动作）？
4. **样本效率**：多轮 Agent 轨迹的采样成本极高，如何提升样本效率？
5. **评估标准**：多步 Agent 任务的评估指标如何设计？（成功率、效率、安全性的权衡）
