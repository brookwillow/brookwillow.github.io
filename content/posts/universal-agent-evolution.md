## 核心论点

通用 Agent 的进化本质上是**人机协作边界的持续后移**：从人主导、模型辅助，到模型主导、人监督，再到模型自主决策、人仅设定目标。每一代产品都在解决上一代遗留的核心瓶颈。

---

## 一、第一代：ChatGPT —— 对话即交互，单轮推理范式

**核心设计思路**
- 交互单元：单次 prompt → 单次 response
- 能力边界：语言理解 + 知识检索 + 文本生成
- Agent 程度：**无** —— 人类是执行者，模型是生成器

**核心瓶颈**
- 无记忆、无状态、无工具
- 每轮对话相互独立，无法完成跨步骤任务
- 用户必须手动分解任务、粘贴中间结果

**历史意义**
> 证明了 LLM 作为自然语言接口的可行性，把"和 AI 对话"从科幻变成日常。

---

## 二、第二代：Manus —— 工具调用 + 多步规划，Agent 范式诞生

**核心设计思路**
- 引入 **Tool Use / Function Calling**：模型不只生成文本，还能调用外部工具
- 引入 **ReAct 范式**（Reason → Act → Observe → Reason...）：把单步推理升级为多步决策循环
- 引入 **任务规划层**：将用户意图分解为可执行子任务

**关键进化**

```
用户意图 → [Plan] → Step1(Tool) → Observe → Step2(Tool) → ... → 最终结果
```

- 沙箱环境：能操作文件、执行代码、访问网络
- 异步执行：任务可以在后台运行，不需要用户实时在场

**核心瓶颈**
- **幻觉执行**：规划看起来合理，但工具调用经常出错，错误会在步骤间累积放大
- **上下文窗口瓶颈**：长任务的中间状态超出上下文限制
- **人介入时机**：不知道什么时候该停下来问人，容易在错误路径上越走越深
- **通用性 vs 专用性矛盾**：工具越多，规划越难

**历史意义**
> 证明了 LLM 可以作为任务执行者而不仅是建议者，"把事情做完"变得可能。

---

## 三、第三代：Claude Code —— 深度工程集成，开发者 Agent 的成熟形态

**核心设计思路**
- **环境感知优先**：不只是调用工具，而是真正"住在"开发环境里（读代码、理解项目结构、运行测试）
- **长上下文 + 持久记忆**：通过 CLAUDE.md、memory 机制跨会话保持项目理解
- **人机协作新模式**：Plan Mode（人批准计划）→ Execute Mode（自主执行）→ 异常时回退询问
- **工具生态标准化**：MCP（Model Context Protocol）把工具接入标准化，降低集成成本

**关键进化**
- **信任层级设计**：不同操作有不同的权限要求（读文件 < 写文件 < 执行命令 < 网络请求）
- **错误恢复能力**：能理解测试失败、编译错误，并自主修复
- **子 Agent 编排**：用 Task 工具启动子 Agent 并行处理独立子任务

**核心瓶颈**
- 仍然局限于**单一领域**（软件开发），通用性不足
- 跨应用、跨系统操作能力弱

**历史意义**
> 第一个在专业领域（软件工程）真正达到"可信赖的协作者"级别的 Agent 产品。

### 三条核心进化主线

**1. 执行自主性提升**
> 人类执行 → 模型建议 → 模型规划+自主执行 → 模型自主完成

**2. 上下文边界扩展**
> 单轮 → 多轮 → 跨会话 → 跨应用 → 跨时间（持久记忆）

**3. 人机信任的精细化**
> 全信任（人执行）→ 全不信任（每步审批）→ 分级信任（按风险审批）→ 目标级信任（只审批意图）

---

## 七、未解决的核心问题

1. **规划幻觉**：Agent 的规划合理性如何保证？形式上可行 ≠ 实际可行
2. **错误放大**：多步执行中，早期错误如何被及时发现和纠正？
3. **意图对齐**：用户说的和想要的往往不一样，Agent 如何处理歧义？
4. **边界设定**：通用 Agent 的"不做什么"比"能做什么"更难定义
5. **可解释性**：当 Agent 做出错误决策，用户如何理解原因？
