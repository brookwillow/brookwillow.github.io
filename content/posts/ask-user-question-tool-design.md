## 核心问题

在 Agent 执行任务的过程中，当遇到信息不足、意图模糊、风险较高的情况时，需要向用户追问、确认或补充信息。

**这个能力应该在哪一层实现？**
- **模型侧**：训练模型在适当时机自主生成问题（自然语言追问）
- **工具侧**：定义标准化的 `AskUserQuestion` 工具，模型通过调用工具来发起追问
- **业务侧**：由上层应用逻辑决定何时追问，模型只负责执行

---

## 一、三种实现方案分析

### 方案 A：纯模型侧（自然语言追问）

**机制**：模型在生成回复时，自然地以文字形式提问，暂停执行等待用户回答。

**优点**
- 零工程成本，无需额外工具定义
- 问题表达更自然、更灵活
- 对用户友好（就像正常对话）

**缺点**
- **结构不透明**：调用方无法程序化地知道"模型在等待用户输入"还是"模型在陈述结果"
- **无法标准化处理**：不同场景的追问混在普通文本中，难以做 UI 差异化
- **缺乏元信息**：无法表达"这个问题是必答的/可选的/有默认值的"
- **追问质量不稳定**：完全依赖模型判断，容易过度追问或遗漏关键问题

**适用场景**：简单对话产品，不需要复杂的追问 UI

---

### 方案 B：工具侧（AskUserQuestion Tool）

**机制**：定义结构化工具，模型决定何时调用，但追问的格式、UI 渲染由工具定义。

```json
{
  "tool": "AskUserQuestion",
  "params": {
    "questions": [
      {
        "question": "你希望优化哪个方向？",
        "header": "优化目标",
        "options": [
          {"label": "性能优先", "description": "减少响应时间"},
          {"label": "成本优先", "description": "减少 token 消耗"}
        ],
        "multiSelect": false
      }
    ]
  }
}
```

**优点**
- **结构化**：调用方可以程序化处理追问状态
- **UI 可定制**：可以渲染成单选、多选、输入框、确认框等不同组件
- **元信息丰富**：可携带选项、描述、是否必填、默认值等
- **可追踪**：系统可以记录追问历史，分析哪些场景模型需要追问
- **可拦截**：业务层可以在追问前做过滤（如某些问题有缓存答案则跳过）

**缺点**
- **模型需要学习何时调用**：训练数据不足时，模型可能在不该问的时候问，或该问时不问
- **格式僵硬**：并非所有追问都适合结构化选项，开放性问题处理不如自然语言灵活
- **工具调用成本**：增加一次工具调用的延迟

**适用场景**：需要精确控制交互流程的 Agent 产品（如 Claude Code、Workflow 工具）

---

### 方案 C：业务侧（应用逻辑控制追问）

**机制**：模型不负责决定是否追问，应用层根据任务类型、缺失参数等规则主动发起追问。

**优点**
- **最高可控性**：追问逻辑完全由产品定义，不受模型能力影响
- **可预测**：同样的情况总是触发同样的追问，用户体验一致

**缺点**
- **覆盖不全**：业务规则无法枚举所有需要追问的场景，复杂任务会有遗漏
- **工程成本高**：需要为每类任务手写追问逻辑
- **丧失 AI 的优势**：模型本身有能力判断"我需要更多信息"，但这套机制完全绕过了它

**适用场景**：高度结构化的表单填写类场景，任务类型固定

## 二、核心设计张力

### 张力 1：灵活性 vs 可控性

自然语言追问灵活但不可控；结构化工具可控但不够灵活。
**解法**：分层设计 —— 结构化工具处理高风险/高频场景，自然语言作为兜底。

### 张力 2：模型自主性 vs 用户体验可预期性

让模型决定何时追问，会导致不同运行的追问时机不一致，用户体验不稳定。
**解法**：在工具层加入"追问策略"元信息，训练模型遵循策略而非完全自主。

### 张力 3：追问的必要性判断

**过度追问**：模型不确定就追问 → 用户体验差，"废话多"
**追问不足**：模型猜测继续执行 → 执行错误，用户需要重来

这是一个**置信度阈值**的校准问题，根本上需要通过训练数据和 RLHF 解决。

## 三、AskUserQuestion 工具的最优设计

### 3.1 何时调用（触发条件）

| 情况 | 示例 | 应对 |
|------|------|------|
| **信息缺失** | "帮我发邮件" 但没有收件人 | 追问必要参数 |
| **意图歧义** | "优化这段代码" 目标不明 | 确认优化方向 |
| **高风险操作** | 删除文件、发布代码 | 显式确认 |
| **多方案并列** | 有多种实现方案各有取舍 | 让用户选择 |

**不应调用的情况**：
- 可以合理推断的信息（上下文已有答案）
- 低风险操作的细节（不需要用户决策）
- 纯粹为了"显得负责"的形式性追问

### 3.2 工具参数设计要点

```typescript
interface Question {
  question: string;         // 问题文本
  header: string;           // 简短标签（UI chip 显示，≤12字）
  options?: Option[];       // 有选项 → 渲染为选择器；无选项 → 文本输入
  multiSelect?: boolean;    // 是否多选
  required?: boolean;       // 是否必填
  defaultValue?: string;    // 默认值
}
```

**关键设计决策**：
1. **批量追问**：一次工具调用支持多个问题，避免分多轮打断用户
2. **有无选项自适应**：选项存在时渲染为结构化选择器，否则降级为文本输入
3. **必填 vs 可选**：用户可以跳过可选问题，模型用默认值或合理推断继续

### 3.3 结论：组合方案最优

- **工具侧**负责：定义追问的格式、UI 渲染、结构化参数
- **模型侧**负责：判断何时需要追问、追问什么（元认知能力）
- **业务侧**负责：特定高风险操作的强制确认（可绕过模型判断直接触发）

## 四、从 Claude Code 的实践中学到的

Claude Code 中的 `AskUserQuestion` 工具是目前最成熟的实现之一：

1. **Preview 功能**：选项可以携带 preview 内容（代码片段、UI 草图），帮助用户直观比较
2. **1-4 个问题的限制**：强制模型精炼追问，不允许一次性抛出大量问题
3. **Other 选项自动添加**：结构化选项不够时，用户可以输入自定义答案

## 五、更深层的问题：追问本身是否应该被优化掉？

一个反直觉的视角：**最好的追问是不需要追问**。

理想的 Agent 应该：
1. 从历史交互中学习用户偏好，减少重复追问
2. 通过上下文推断，自动填充大多数缺失信息
3. 只在真正需要用户**决策**（而非信息补充）时才追问

追问工具的长期演化方向是：**用得越来越少，但每次用都更有价值**。
